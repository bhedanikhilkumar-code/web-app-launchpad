# Review Checklist — Web App Launchpad

Use this checklist before sharing the repository with recruiters, reviewers, collaborators, or users.

## First Impression

- [ ] Repository description and topics are accurate.
- [ ] README title, badges, and overview are polished.
- [ ] The project value is understandable in the first 30 seconds.
- [ ] Screenshots, demo GIFs, or placeholders are present where useful.

## Documentation Depth

- [ ] README links to the documentation hub.
- [ ] `docs/ARCHITECTURE.md` explains layers and workflow.
- [ ] `docs/CASE_STUDY.md` explains product thinking and decisions.
- [ ] `docs/ROADMAP.md` has realistic next steps.
- [ ] `docs/QUALITY.md` explains checks and repository standards.

## Engineering Signal

- [ ] Project structure is easy to navigate.
- [ ] Setup commands are documented.
- [ ] Known limitations are documented honestly.
- [ ] No secrets, private keys, or real credentials are committed.
- [ ] Generated/binary files are intentional and documented.

## GitHub Professionalism

- [ ] Issue templates are present.
- [ ] Pull request template is present.
- [ ] CONTRIBUTING, SECURITY, SUPPORT, and CODE_OF_CONDUCT files exist.
- [ ] CODEOWNERS is set.
- [ ] Repository Health workflow passes.

## Portfolio Demo Script

When presenting this project, use a simple narrative:

1. **Problem:** What pain point or use case it targets.
2. **Solution:** What the project does.
3. **Architecture:** How the main pieces connect.
4. **Workflow:** The key user journey.
5. **Quality:** How it is documented, checked, and maintained.
6. **Roadmap:** What would make it production-grade next.
